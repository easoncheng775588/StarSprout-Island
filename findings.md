## P0 Settlement Loop 2026-04-20

- 当前掌握规则先采用 `correctCount >= 3 && wrongCount == 0`，便于孩子端一键复习闭环；后续可演进为按知识点连续多次正确、间隔复习和遗忘曲线
- P1 更适合沿现有体验面继续增强：通关庆祝与榜单反馈、家长端多孩子对比、首页今日任务状态化

- 展示顺序采用当前孩子置顶，其余孩子按本周星星和完成关数排序，比直接排名更温和，也更符合儿童学习产品的语气

- 幼小衔接数学已有数数、20 以内加减、规律、比较、列式和故事题，但缺图形分类、方位理解、序数/第几个这类入学前空间与数感内容

- 幼小衔接英语已有 26 字母分组、开头音、生活词、颜色词和 3 个绘本，但缺大小写配对、家庭生活词和更多日常绘本场景
- 本轮适合新增 9 个小关卡，每科 3 个，复用现有互动玩法，避免先引入全新复杂玩法导致测试和内容配置膨胀

## Grade-by-Grade Productization 2026-04-22

- 当前最主要的代码质量风险在前端 `LevelPlayer.tsx`：多种玩法直接内联在页面中，继续扩关卡会让页面越来越难维护
- 课程数据库结构目前仍然够用，逐年级产品化不需要先新增内容表；新增关卡可以继续用稳定 code 写入 `levels` 和 `level_steps`
- `level_steps.id` 当前是技术主键，业务引用仍使用 `level_code + step_code`，后续内容不要依赖自增 ID 做业务含义
- 第一阶段最稳妥的工程顺序是：先抽可复用互动组件，再补幼小衔接成熟化内容，再推进一年级到四年级
- 成熟产品感不只来自关卡数量，还来自统一关卡流程、讲解、反馈、复习、家长端报告和成就闭环
- 音频控制在听辨、跟读、短句朗读中重复出现，适合先抽为 `AudioModeControls`，后续再继续拆分选项卡、反馈卡和媒体模型组件
- 幼小衔接数学补充“快速看数量、数的分合、看图加法”比单纯增加算式更符合入学准备阶段的数感目标
- 一年级数学继续扩展时，生活应用比单纯增加计算题更能形成正式产品路径；人民币、整点时间、图形拼组都可以先复用现有关卡模型并用后端配置驱动
- 一年级语文/英语成熟化适合围绕“听辨 -> 搭配 -> 跟读表达”形成小闭环，现有 `listen-choice`、`character-choice`、`word-match`、`sentence-read` 已能覆盖第一批内容
- 二年级成熟化应从“能算”推进到“能解释结构”：平均分、统计图、中心句和主题对话都适合做成看模型、做选择、听反馈的短闭环
- 三四年级继续成熟化时，不宜先引入开放作文、语音评分或几何作图；这些会显著增加评分规则和测试成本，当前更适合复用现有玩法，把知识点覆盖和反馈质量先做扎实
- 三年级成熟化的关键是从单点题目转向“模型和语言表达”：有余数除法、面积、概括主要内容、一般疑问句都能用现有短交互承载，不需要先做复杂编辑器
- 四年级成熟化适合强化“抽象概念可视化 + 阅读线索判断 + 场景表达”：小数比较、平行垂直、古诗意象和时态关键词都能让产品更接近正式课程路径
- 题库配置中心不能只列“已经配置好的关卡”，还要把缺口显性化；覆盖率、健康/预警数量和逐关卡缺口说明更适合后续运营补素材、补音频和补玩法配置
- 成就系统继续成熟化时不宜直接改动全局徽章统计口径；新增学段成就家族作为并行结构，可以保留首页/通关奖励兼容性，同时让孩子在当前学段看到更明确的成长目标
- 音频体验优化不应继续把浏览器 TTS 直接写在玩法页面里；先抽象为播放计划和播放结果，后续接入录音素材或第三方 TTS 时可以只替换音频层，互动玩法保持稳定
- 家长周报不需要先生成 PDF 文件；先把周报数据结构化并做成可打印报告卡，可以复用同一份数据继续接 PDF、分享海报或家长端导出
- 周报导出体验应先通过 CSS 打印样式完成单页可读性，避免过早引入 PDF 生成链路；后续可复用同一周报数据做 PDF/分享海报
- 家长端薄弱点不应停留在“提醒”层；用错题记录和知识点标签生成陪练话术、复习动作和直达关卡，能把报告转化成家长当天可执行的陪伴动作
- 数形结合关卡需要在家长端沉淀为“模型能力”而不是散落的单关记录；先用知识点 code 和 `mathModel` 配置动态归类，能展示数轴、线段图、面积模型、分数条等成长轨迹
- 孩子端成就墙也需要承接模型能力，否则模型训练只被家长看见；模型徽章可以先实时计算，保持数据库简洁，同时让孩子获得明确的模型掌握反馈
- `LevelPlayer.tsx` 已经承载了太多互动细节，数形结合模型是最适合优先抽离的模块；它没有关卡提交副作用，抽成纯组件后能同时服务数学岛和未来奥数模型题
- 图片算式在幼小衔接和低年级数学里会高频出现，抽成组件后可以统一控制儿童视觉密度、可访问标签和后续素材替换，不必每个关卡重复维护

## Mathseeds-Inspired Product Maturity 2026-04-23

- Mathseeds 的核心可借鉴点不是单个游戏，而是“能力诊断 -> 地图路径 -> 结构化小课 -> 测评/奖励 -> 家长报告”的完整循环
- 当前产品已经有学段、学习路径、成就、家长报告和错题闭环，P0 成熟化最适合先补“能力测评入口”，让孩子不只是手动选年级，而是由轻量诊断推荐起点
- 能力测评 MVP 不需要新增数据库表，推荐结果可以先写回 `child_profiles.stage_label`；后续如果需要记录测评历史、题目明细和多次成长曲线，再考虑独立 assessment 表
- 结构化小课不应直接堆在 `LevelPlayer.tsx` 里；抽成 `LessonFlowPanel` 后，播放器只传入步骤进度和奖励状态，后续要扩展证书、小测或复盘说明时更容易复用
- 第一版小课外壳适合只覆盖数学岛和奥数板块，符合当前 P0“数学成熟化”的优先级；语文/英语可在后续 P1/P2 做各自的听说读写学习流程
- 阶段证书适合先成为成就墙里的“证书夹”，由现有学段徽章进度生成；这能提升正式产品感，同时避免过早引入证书编号、签发记录等数据复杂度
- 每日 1 分钟快练适合先做短平快的数感入口，而不是一开始就做复杂计时赛；先保证学段题组、即时反馈和完成感，再持久化成绩并接入连续训练趋势
- `fluency_attempts` 是有明确业务归属的独立表：它记录每日短练这种非关卡型练习，不能混入 `level_completions`，否则会污染主线关卡完成语义
- 家长端“数感快练趋势”优先用独立卡片承接，比现在就塞进周报更稳；这样既能快速让新持久化数据产生用户价值，也能避免周报 DTO 在成熟化阶段过早膨胀
- 数感快练进入可视化阶段后，最合适的接口边界是“后端返回最近 7 天轻量点位，前端只负责展示”；P2 再在此基础上增加按学段或题型的深一层洞察，而不是让前端自行聚合历史记录
- P2 的第一步更适合先做“按学段”的快练分层洞察，而不是直接做题型分析：现有 `fluency_attempts` 已天然带学段标签，能稳定产出陪练建议；题型洞察应等快练题目结构进一步标准化后再补

## Fluency Type Insights 2026-04-23

- 家长端题型洞察和周报联动复用同一份后端聚合结果，避免前端重复推导状态

- 音频体验升级的最小可行路径是继续复用 `activityConfigJson`，由前端音频层支持 `recordedAssetUrl` 优先和播放失败时的 TTS 兜底，不必先引入独立素材表
- 练习强度偏好挂在现有 `parent_settings` 最稳，因为当前设置本来就是按孩子存储；这样首页与每日任务文案可以直接联动，不会扩大孩子档案写接口的改动面
- 奥数训练营当前的主要短板不是“没有年级覆盖”，而是“后端地图没有显式返回全部 1-6 年级、课程路线表达不够统一”；先把学科地图和小课表达标准化，收益大于继续盲加关卡
- 发布前检查目前最有效的投入仍是扩充 smoke test 与演示文档，而不是过早上复杂脚本；全量前后端验证已经可以覆盖当前 MVP 到产品化阶段的大部分回归风险

## Content Config Editor MVP 2026-04-23

- 题库配置中心已经具备“目录 -> 详情 -> 保存 -> 健康反馈”的最小可用链路，适合作为后续 CMS 化之前的运营入口
- 配置详情与配置目录共用一套后端配置快照逻辑，可以降低 `assetTheme`、`audioQuality`、`healthStatus` 在不同页面计算不一致的风险
- 配置中心展示名更适合使用 `levels.detail_title` 而不是 `summary_title`，因为它更贴近运营/内容命名，也和关卡详情页的儿童化标题一致
- 当前配置保存仍是“直接改 JSON 文本”，适合 MVP 快速推进；进入 Phase 52/53 后如果要扩大运营使用范围，建议把高频字段拆成结构化表单，JSON 仅作为高级配置区
