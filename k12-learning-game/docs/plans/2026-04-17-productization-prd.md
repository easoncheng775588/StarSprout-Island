# K12 Learning Game V1.1 产品化升级 PRD

## 1. 文档信息

- 产品名称：Learning Island 幼小衔接学习游戏
- 当前版本：V1 MVP
- 目标版本：V1.1 产品化升级版
- 文档日期：2026-04-17
- 适用范围：当前 React + Java + MySQL 单体前后端项目

## 2. 背景与现状

当前版本已经完成幼小衔接阶段的核心学习闭环：

- 数学岛、语文岛、英语岛基础关卡可游玩
- 关卡完成后可获得星星与奖励反馈
- 已有首页、学科地图、成就墙、排行榜、家长中心
- 已有基础账号态：孩子档案选择、登录态本地持久化、孩子切换
- 已有后端持久化：关卡完成记录、部分成长数据、动态成就与榜单计算

但当前整体仍更偏“可演示 MVP”，距离“正式产品体验”还有几个明显缺口：

- 家长端可看但不可配，缺少真正生效的家长控制
- 排行榜已有动态数据，但还缺真实参与资格、结算口径和隐私策略闭环
- 成就系统已有规则，但缺少层级、展示节奏、解锁反馈和家长解释性
- 账号体系仍偏轻量，缺少正式的家长账号与孩子档案管理
- 关卡进度虽然已入库，但前后端联动、实时刷新和异常兜底还不够完整
- 音频体验仍依赖系统 TTS，缺少更像正式产品的语音偏好与资源管理能力

## 3. 目标与推荐方案

### 3.1 本阶段目标

V1.1 的核心目标不是继续横向铺大量新内容，而是把现有幼小衔接内容打磨成“可持续扩展的正式产品底座”。

本阶段重点解决四件事：

1. 做出正式产品感的账号与档案体系
2. 让学习进度、成就、排行榜、家长端形成真实联动
3. 让家长端从“报告页”升级成“可管理、可配置、可陪伴”
4. 把音频与关卡反馈从“能用”升级到“更顺手、更稳定、更像儿童产品”

### 3.2 备选方案

方案 A：继续扩关卡和内容
- 优点：短期看起来更丰富
- 缺点：产品底座不稳，后续返工大，家长端和账号体系会拖住增长

方案 B：优先做产品化闭环，再继续扩内容
- 优点：后续每加内容都能直接复用账号、进度、成就、家长体系
- 缺点：短期新增可玩内容数量不如继续铺关卡显眼

方案 C：同时推进内容和产品化
- 优点：两边都能有进展
- 缺点：当前项目体量下容易摊薄资源，质量不稳定

### 3.3 推荐结论

推荐采用方案 B：先完成产品化闭环，再继续扩更多学科内容与关卡。

## 4. 用户与核心场景

### 4.1 目标用户

- 儿童用户：幼小衔接阶段 5-7 岁儿童
- 家长用户：陪伴学习、查看成长、控制使用时长与激励策略的家长

### 4.2 核心场景

#### 儿童侧

- 进入应用后快速回到自己的学习进度
- 完成今天推荐任务并获得即时鼓励
- 清楚知道下一关是什么、为什么推荐这关
- 在完成关卡后立即看到成就、星星和榜单变化

#### 家长侧

- 登录后看到多个孩子档案并切换查看
- 看到今天学了什么、学得怎么样、哪里需要陪练
- 配置每日学习时长、是否参与排行榜、是否开启连续学习提醒
- 明确理解榜单和成就为什么发生变化

## 5. 版本范围

### 5.1 P0 必做范围

#### 模块一：正式账号与档案体系

目标：把当前“孩子直接进入”升级成“家长账号 + 孩子档案”的正式产品结构。

功能点：

- 家长账号登录页
- 家长账号注册入口（MVP 可为手机号验证码的静态模拟流程，先不接真实短信）
- 家长账号下创建/编辑孩子档案
- 每个孩子档案支持：
  - 昵称
  - 年龄/学段
  - 头像或头像颜色
  - 当前阶段标签（幼小衔接）
- 登录后默认进入“上次活跃孩子”
- 顶部支持切换孩子
- 退出登录后清除本地会话

验收标准：

- 同一家庭下可维护 1-3 个孩子档案
- 切换孩子后首页、学科地图、家长端、成就墙、排行榜全部切换到对应数据
- 刷新页面后仍能保持当前登录态和当前孩子

#### 模块二：进度持久化与联动升级

目标：让学习记录真正成为系统底层事实，前后端展示一致。

功能点：

- 关卡完成记录去重规则
- 首次通关、重复练习区分
- 每关保存：
  - 完成时间
  - 正确数
  - 错误数
  - 用时
  - 是否首次完成
- 首页推荐任务基于未完成主线
- 学科地图状态分为：
  - 已完成
  - 推荐挑战
  - 可挑战
- 完成关卡后以下页面刷新一致：
  - 首页
  - 学科地图
  - 成就墙
  - 排行榜
  - 家长中心

验收标准：

- 任意关卡完成后返回首页，星星、成就预览、推荐任务立即正确变化
- 重复完成同一关时，不重复计算“首次解锁类”成就
- 数据刷新以接口真实返回为准，不依赖前端写死推算

#### 模块三：排行榜真实结算逻辑 V2

目标：把排行榜从“展示页”升级成“有规则、有边界、有隐私控制”的正式系统。

榜单范围：

- 本周星星榜
- 连续学习榜
- 挑战达人榜

核心规则：

- 本周星星榜：
  - 只统计近 7 天有效学习获得的星星
  - 重复练习同一关可计入练习次数，但星星收益需有上限策略
- 连续学习榜：
  - 统计连续自然日内是否完成至少 1 次有效学习
- 挑战达人榜：
  - 统计近 7 天有效完成关卡次数

配套策略：

- 家长可关闭排行榜参与
- 关闭后：
  - 该孩子不出现在公开榜单
  - 自身页面展示“未参与排行榜”
- 榜单默认用昵称展示，后续预留匿名昵称策略
- 排名并列时按最近活跃时间优先，再按档案创建时间排序

验收标准：

- 切换不同孩子时，榜单排名与目标差距提示正确变化
- 家长关闭榜单参与后，该孩子不进入榜单排序
- 页面展示结算周期、更新时间、距离上一名差距

#### 模块四：成就系统 V2

目标：让成就系统既能激励孩子，又能让家长理解成长路径。

成就分类：

- 数学启蒙
- 语文启蒙
- 英语启蒙
- 学习习惯
- 每周挑战
- 综合朗读

规则层级：

- 首次解锁类：如完成首个数学关卡
- 累积数量类：如完成 5 个数学关卡
- 连续学习类：如连续学习 3 天、7 天、10 天
- 准确率类：如平均准确率达到 90%
- 学习时长类：如累计学习 40 分钟

展示要求：

- 每枚成就显示：
  - 名称
  - 分类
  - 稀有度
  - 当前进度
  - 解锁文案
  - 鼓励文案
- 新解锁成就在关卡完成页弹出庆祝反馈
- 家长端展示“下一枚最接近解锁的成就”

验收标准：

- 已解锁与进行中分组清晰
- 进行中成就按接近解锁程度排序
- 首次通关类成就不会重复解锁

#### 模块五：家长端 V2

目标：让家长端从“查看结果”升级成“查看 + 配置 + 陪伴决策”。

数据维度：

- 今日完成关卡数
- 今日学习分钟数
- 今日获得星星数
- 最近 7 天学习趋势
- 学科进度
- 学科拆解：
  - 已完成关卡数
  - 总关卡数
  - 学科准确率
  - 学科学习分钟数
  - 推荐下一关
- 学习画像：
  - 平均准确率
  - 平均单关时长
  - 最佳学习时段
  - 当前优势学科
- 最近活动时间线
- 薄弱点建议

配置能力：

- 设置每日目标学习时长
- 是否参与排行榜
- 是否开启连续学习提醒（先做前端开关与后端持久化，提醒发送后置）

验收标准：

- 配置修改后刷新仍保留
- 关闭排行榜后榜单接口与页面表现同步变化
- 家长端能解释“为什么推荐下一关”和“为什么该学科是薄弱点”

#### 模块六：音频体验与素材质量 V2

目标：让朗读体验更稳定、更儿童化，并为后续真音频资源接入预留结构。

功能点：

- 系统 TTS 继续保留
- 支持语速偏好持久化
- 显示当前音色/语音老师
- 绘本关支持整段领读
- 设备不支持语音时优雅降级
- 后端与前端预留素材字段：
  - audioUrl
  - voiceType
  - sourceType（system_tts / uploaded_audio）

验收标准：

- 刷新后仍记住上一次语速模式
- 朗读按钮、跟读按钮、整段领读按钮逻辑一致
- 没有语音能力的环境不会报错

### 5.2 P1 应做范围

- 关卡完成后的“成就解锁动画 + 榜单上升提示”整合
- 孩子档案头像库
- 首页“今日任务卡”支持完成状态和连续打卡反馈
- 家长端增加孩子切换器与多孩子横向对比

### 5.3 P2 后续范围

- 真短信/邮箱登录
- 云端音频资源上传与分发
- 绘本音频分句对齐
- 推送提醒
- 更多年级内容扩展

## 6. 非目标

本阶段不做：

- 支付、订阅、课程购买
- 社区互动、好友关系
- 教师端
- 真实直播课或 AI 对话老师
- 小学 1-6 年级全部内容铺开

## 7. 关键业务规则

### 7.1 有效学习定义

一次关卡完成满足以下条件，才记为有效学习：

- 完成全部步骤
- 用时超过最小阈值
- 正确率不低于基础阈值，或明确完成所有必做步骤

### 7.2 重复练习规则

- 首次完成：计入首次完成、主线进度、首次解锁成就
- 重复完成：计入复习次数、时长、挑战次数
- 重复完成同一关的星星奖励需要做上限限制

推荐 MVP 规则：

- 同一关首次完成给满星
- 同一自然日重复练习只计学习行为，不重复加满星

### 7.3 连续学习规则

- 每个自然日完成至少一次有效学习则记为连续 1 天
- 若某一天没有有效学习，则连续天数断开

## 8. 数据与接口需求

### 8.1 需要新增/补强的数据模型

- parent_accounts
- user_sessions
- child_profiles（补充 avatar、birth_year/age_stage、last_active）
- level_completions（补充 is_first_completion、is_effective）
- parent_settings（补充 reminder_enabled）
- achievement_unlocks（可选；若本阶段继续动态计算，可先不建表）

### 8.2 需要新增/补强的接口

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/session`
- `GET /api/parent/children`
- `POST /api/parent/children`
- `PATCH /api/parent/children/{id}`
- `PATCH /api/parent/settings`
- `GET /api/home/overview`
- `GET /api/parent/dashboard`
- `GET /api/achievements`
- `GET /api/leaderboard/{boardType}`
- `POST /api/levels/{levelCode}/complete`

## 9. 交互与体验原则

- 儿童操作路径不超过 2 次点击进入关卡
- 成就、星星、下一关提示要即时可见
- 家长配置项文案要简单，不用专业术语
- 排行榜强调“成长比较”，弱化竞争焦虑
- 所有数据类页面都要有空态、加载态、失败态

## 10. 技术实现建议

### 10.1 前端

- React 保持当前路由结构
- 统一会话上下文，接管登录态与当前孩子
- 所有依赖孩子身份的页面统一监听当前孩子变化
- 关卡完成后使用统一的 refresh 流程更新首页、榜单、成就、家长端缓存

### 10.2 后端

- Java Spring Boot 继续作为主服务
- 使用 MySQL/H2 双环境兼容
- 以 `level_completions` 作为成长数据的核心事实表
- 以服务层统一计算首页、成就、排行榜、家长报告，避免前端重复推算

## 11. 验收标准

### 11.1 产品验收

- 家长可登录并管理孩子档案
- 切换孩子后所有页面数据同步变化
- 关卡完成后首页、学科地图、成就、榜单、家长端表现一致
- 家长端设置项可保存并真实生效
- 榜单关闭参与后页面和数据逻辑正确
- 新成就解锁与进行中成就排序符合规则

### 11.2 技术验收

- 前端新增与更新页面有自动化测试
- 后端新增规则有接口测试与服务测试
- 不破坏现有幼小衔接关卡体验
- 前后端本地联调稳定
- 刷新、切换孩子、退出登录、重复练习等关键流程通过回归测试

## 12. 开发顺序建议

推荐按以下顺序执行：

1. 家长账号与孩子档案模型
2. 会话体系与登录态改造
3. 家长设置持久化
4. 关卡完成记录规则补强
5. 排行榜真实参与与结算逻辑
6. 成就系统规则升级与完成页反馈
7. 家长端 V2 展示与配置联动
8. 音频体验和资源字段预留
9. 全链路联调与自动化测试补完

## 13. 风险与应对

- 风险：账号体系一上来做太重，开发周期拉长
  - 应对：V1.1 先做“轻量正式登录”，不接真实短信服务
- 风险：成就与榜单规则复杂，前后端口径不一致
  - 应对：统一以后端规则为准，前端只展示
- 风险：音频体验依赖浏览器系统能力
  - 应对：保留 TTS 降级方案，并预留上传音频结构
- 风险：现有测试覆盖不足，回归容易漏
  - 应对：按模块补自动化测试后再集成

## 14. 结论

V1.1 的本质不是“多做几关”，而是把幼小衔接现有内容升级成真正能持续迭代的产品底座。

这版 PRD 确认后，建议直接进入开发，按“账号与档案 -> 进度与规则 -> 家长端与体验 -> 联调与测试”的顺序推进。
